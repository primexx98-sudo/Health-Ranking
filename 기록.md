## 월별 집계 점수 공식 — 결정 과정 (2026-07-01 인터뷰)

**기존 문제:** 평균순위 1위인데 등장횟수 1~2번인 제품이 월간 TOP10 상위에 오는 정확도 문제

| 항목 | 결정 |
|------|------|
| 처리 방식 | B) 순위 하향 — 점수 공식으로 등장 적은 상품을 자연스럽게 하위에 배치 |
| 점수 체계 | 1위=5점, 2위=4.5점 … 10위=0.5점 (0.5점 차등) |
| 등장일 처리 | 미등장일 0점, 실제 수집일수 기준 등장률 산정 |
| 가중치 | 순위 70% + 꾸준함 30% |
| 출력 컬럼 | 기존 유지 (평균순위 표시, 점수 컬럼 없음) |

**시행착오**
1. 첫 시도: `월간점수 = 평균점수 × 등장률` (분모=달력 30일) → 등장률이 너무 낮아 등장 적은 제품 과도 패널티
2. 분모를 실제 수집일수로 변경 → 개선
3. 가중치 50:50 시도 → 등장 많은 저순위 제품이 1위 제품을 역전하는 문제
4. 가중치 70:30 확정 → 순위 질 우선, 꾸준함 보조 반영

**2026-07-02 — 최소 등장횟수 필터 추가**
등장횟수 3회 미만 상품은 월간 랭킹 대상에서 제외 (1~2회만 등장해도 평균순위가 좋으면 상위권에 오르는 문제 방지). 다이소몰처럼 표본이 적은 플랫폼에서 3회 필터만 적용하면 TOP10이 안 채워지는 문제가 발견되어, 필터 후 10개 미만이면 최소등장 기준을 2회 → 1회로 순차 완화하도록 추가 수정.

## 올리브영 크롤러 curl_cffi → Playwright 전환 (2026-07-13 인터뷰/의사결정)

**증상**: 7/11, 7/12, 7/13 Daily Crawl 3회 연속 실패. 로그 확인 결과 `crawl_oliveyoung()`의 재시도 3회가 매번 즉시 HTTP 403 — "간헐적 403"이 아니라 "고정 차단" 패턴으로 전환됨.

**원인 진단**:
1. 동일 코드(`impersonate="chrome146"`, 동일 헤더)를 로컬(가정용 IP)에서 그대로 실행 → 200 정상 응답, 상품 정상 파싱
2. 사용자가 시크릿모드로 접속해도 올리브영 정상 접속 확인
3. → TLS 핑거프린트/impersonate 버전 문제가 아니라 **GitHub Actions(Azure 데이터센터) IP 자체가 Cloudflare에 평판 기반으로 차단**된 것으로 결론

**대응 옵션 검토** (1차 인터뷰):
- A) 유료 프록시/스크래핑 API 연동
- B) 무료 프록시(Webshare 무료 티어)
- C) 올리브영만 로컬 PC로 분리 수집
- D) Self-hosted Actions runner로 전체 이전

→ 1차로 **B) 무료 프록시**를 선택했으나, 착수 직전 사용자가 참고 사례(the-underdogs.kr 올리브영 경쟁사 모니터링 자동화 글, Selenium+GitHub Actions로 76일 무장애 운영)를 제시.

**재검토**: 해당 사례는 curl_cffi가 아닌 **실제 헤드리스 브라우저(Selenium)** 로 같은 Actions 환경에서 안정 운영 중 — curl_cffi는 JS를 실행하지 않아 Cloudflare의 JS 챌린지 자체를 통과할 수 없는 반면, 실제 브라우저는 통과 가능하다는 가설이 성립. 이미 카카오·다이소가 Playwright로 같은 Actions IP에서 문제없이 동작 중이라는 사실도 이 가설을 뒷받침.

**최종 결정**: 프록시 계정 신설 없이, 올리브영 크롤러를 curl_cffi+BeautifulSoup → **Playwright(카카오·다이소와 동일 패턴)** 로 전환. 셀렉터는 기존 `ul.cate_prd_list > li` / `span.tx_brand` / `p.tx_name` 그대로 유지(서버 렌더링 HTML이라 셀렉터 자체는 안 바뀜). 로컬 테스트에서 10개 정상 수집 확인. `requirements.txt`에서 `curl_cffi`/`beautifulsoup4`/`lxml` 제거.

**1차 push 후 즉시 workflow_dispatch로 검증** (익일까지 기다리지 않음): 403은 사라졌으나 `ul.cate_prd_list` 셀렉터가 빈 값(`no_products`)으로 3회 재시도 모두 실패 — 헤드리스 브라우저 자동화 흔적(`navigator.webdriver` 등)을 Cloudflare가 탐지해 다른 페이지를 반환한 것으로 추정.

**2차 대응**: `requirements.txt`에 이미 있었지만 실제 코드 어디에도 적용되지 않고 있던 `playwright-stealth`를 발견 (버전 1.0.6 → 로컬 실설치 2.0.3에 맞춰 갱신). `crawlers/base.py`의 `new_page()`에 `Stealth().apply_stealth_sync(context)` 추가해 전체 크롤러(카카오·다이소·올리브영)에 공통 적용. 카카오는 stealth 적용 후에도 로컬 재검증에서 회귀 없음(20개 정상 수집) 확인 후 재push, Actions에서 재검증.

## 올리브영 Turnstile 원인 확정 및 캡쳐 기반 수동 입력 채택 (2026-07-13, 후속)

**2차 대응(stealth 적용) 재검증 결과**: Actions에서 여전히 `no_products` — 403이 아니라 실패 스크린샷을 열어보니 Cloudflare **Turnstile 인터랙티브 체크박스**("사람인지 확인하십시오")가 뜨고 있었음. 단순 JS 챌린지가 아니라 사람의 클릭을 요구하는 챌린지로, 지금까지의 진단(403 → 헤드리스 탐지)보다 한 단계 더 강한 방어였다는 게 확인됨.

**추가로 시도했으나 전부 실패한 대응**:
1. **stealth 완전 제거 후 순정 Chromium 재테스트** — 동일하게 막힘. 다이소·카카오는 stealth 없이도 정상 동작. → 브라우저 지문(fingerprint) 문제가 아님을 확인 (커밋 `57d8d6b`)
2. **Turnstile 체크박스 Playwright 자동 클릭 시도** — `page.frame_locator('iframe[src*="challenges.cloudflare.com"]')`로 iframe 자체는 잡히지만, 그 안 `body.innerHTML`이 항상 빈 문자열로 반환됨(4초 대기 후에도 동일). 실제 체크박스 UI가 일반 DOM이 아닌 접근 불가능한 렌더링 방식(shadow DOM 등 추정)으로 그려져 있어 셀렉터/클릭 자체가 도달 불가 — Cloudflare가 스크립트 자동화를 막기 위해 의도한 설계로 판단 (커밋 `e65422a`, 디버그 커밋 `1bc26ca`/`ccb10e8`/`3654ba1`, 최종 제거는 `c2a96e9`)
3. **IP 랜덤 재시도** — workflow_dispatch를 동시에 4회 실행(각각 다른 Azure IP 할당) → **4개 전부 동일하게 실패**. 오늘 하루 총 9개 IP에서 테스트해 전부 같은 패턴 → 개별 IP 평판이 아니라 **Azure 데이터센터 IP 대역(ASN) 자체**를 Cloudflare가 무조건 차단하는 것으로 결론. 재시도 횟수를 늘려도 통과 확률은 사실상 0
4. **사용자 제공 `requests` 기반 스크립트(Accept/Referer 헤더만 추가) 검토** — 로컬(가정용 IP)에서 직접 재현 테스트한 결과 동일하게 403. `requests` 라이브러리의 TLS 지문(JA3)이 브라우저가 아님을 바로 드러내는 게 원인으로 추정 — 헤더 조합이 아니라 TLS 핸드셰이크 지문이 판별 기준이었던 것으로 보임 (IP 문제와는 별개의 원인)

**최종 결론 (코드/IP 레벨 대응 종료)**: 무료로 시도 가능한 대응은 모두 소진. self-hosted runner도 검토했으나 PC가 평소 수집 스케줄(09:00) 시간대에 꺼져 있어 기각. 유료 프록시 외에는 안정적인 완전 자동화 방법이 없다는 데 도달.

**채택한 대안**: 카카오·다이소는 지금처럼 100% 자동 유지. 올리브영은 기존 자동 시도(3회 재시도)+실패 시 카카오톡 알림 흐름을 그대로 두되, 실패 시 **사용자가 랭킹 페이지를 직접 캡쳐해서 Claude에게 전달 → Claude가 이미지에서 TOP10을 읽어(비전 인식) `manual_oliveyoung.py`로 그날 xlsx에 반영**하는 반자동 절차를 추가. 상품URL은 캡쳐에 안 보여 공란 처리. 실제 캡쳐로 10개 전량 반영 및 카테고리통계 재계산 검증 완료(커밋 `38cc1dc`). 이 절차를 매번 설명할 필요 없이 새 대화에서 바로 수행할 수 있도록 실행 매뉴얼 `크롤러_실패시.md`를 별도로 분리(커밋 `1f7acb9`).
